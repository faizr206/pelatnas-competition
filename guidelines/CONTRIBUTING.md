# Contributing

## Branching
- `main` is protected
- use feature branches
- prefer small pull requests

## Code style
- keep modules focused
- avoid framework-specific logic in domain packages
- write tests for bug fixes
- document non-obvious design decisions

## Pull request checklist
- tests added or updated
- migrations reviewed
- API changes documented
- docs updated if architecture or workflow changed
- security impact considered

## Commit style
Use clear commit messages, for example:
- `feat(api): add submission status endpoint`
- `fix(worker): enforce timeout cleanup`
- `docs: add execution sandbox rules`
